## Design decisions (made before first commit)

- Core: `containment/plugin/units/colorizerdecision.h`, header-only
  pure functions in `Latte::Containment::ColorizerDecision`:
  - `chooseThemeSource(env) -> ThemeSource` - the applyTheme decision
    tree (Manager.qml 68-136) as an enum-returning function. The enum
    names WHICH live object applies (PlasmaDefault, the two
    active-window scheme variants, TouchingWindow, Dark/Light theme,
    LayoutScheme); mapping enum -> QObject stays in the wrapper.
  - `appliedSchemeIsPlasmaTheme(source, env) -> bool` - the object
    identity algebra behind every `applyTheme === plasmaTheme`
    comparison, made explicit. Verified against the C++ providers:
    Theme::darkTheme() returns m_defaultScheme iff !isLightTheme,
    lightTheme() iff isLightTheme (app/plasma/extended/theme.cpp
    180-187), so Dark aliases the default exactly when the theme is
    dark and Light when it is light. Layout schemes come from
    wm Schemes::schemeForFile (own instance map,
    app/wm/tracker/schemes.cpp) and window schemes from the same
    tracker map - never identity-equal to Theme's m_defaultScheme, by
    construction. Recorded in the header comment.
  - `mustBeShown(env, source, appliedThemeExists) -> bool` -
    Manager.qml 58-59. appliedThemeExists is the wrapper's
    resolved-pointer non-nullness (null resolution is live state, so
    it crosses the boundary as a fact, not a guess).
  - `chooseSchemeFile(env, source, appliedThemeExists) -> SchemeFile`
    - the scheme string selection (Manager.qml 167-199) as an enum
    over {AppliedScheme, LightThemeFile, DarkThemeFile,
    DefaultThemeFile, KdeGlobalsFallback}. The Qt5 body's
    `darkTheme === defaultTheme` test is the aliasing algebra above
    (== !plasmaThemeIsLight), kept in that shape with a comment.
  - `useLayoutTextColor(env) -> bool` - the edit-mode text color
    override (Manager.qml 141-151: configure mode + compositing +
    stored opacity < 0.40 + SmartThemeColors + layout exists).
- Env struct: plain values only. themeColors/windowColors use the
  plugin's REAL Q_ENUM types (Types::ThemeColorsGroup,
  Types::WindowColorsGroup from containment/plugin/types.h) - no
  mirror enums, no drift, per the assert-early law's
  compiler-enforced-sync preference.
- Wrapper: `ColorizerDecider` QObject in containment/plugin/
  (colorizerdecider.h/.cpp), registered creatable in
  org.kde.latte.private.containment like AutoSizeStepper. Declarative
  property shape (the IconColorfulness/EX-04 idiom): input
  Q_PROPERTYs for the env scalars plus QPointer-held QObject* inputs
  for the seven candidate scheme objects; outputs applyTheme
  (resolved QObject*), schemeColors (the object whose schemeFile
  applies, null => "kdeglobals"), mustBeShown, useLayoutTextColor.
  The wrapper derives the schemeExists env bools from its stored
  QPointers and passes resolved non-nullness back into
  mustBeShown/chooseSchemeFile - all DECISIONS stay in the core;
  the wrapper only snapshots pointers and maps enums to members.
  QPointer guards the deleteLater window when Theme rebuilds its
  SchemeColors (theme.cpp 233-238); the QML rebind on themeChanged
  refreshes the inputs as before.
- QML keeps (Manager.qml stays the authority for): plasmaTheme
  derivation, all colorBrightness math (EX-19's turf), outlineColor,
  editModeTextColor, applyingWindowColors (lines 63-66; outside the
  spec's named ranges and consumed only through the bridge surface),
  the Loader/BackgroundTracker wiring, and the Kirigami fallbacks for
  a null applyTheme. `scheme` maps
  `decider.schemeColors ? .schemeFile : "kdeglobals"`.
- The scheme string and applyTheme object are PUBLIC API surface:
  LatteBridge.qml exposes colorizerManager wholesale as colorPalette
  to latte-aware client applets, and the shell settings canvas reads
  latteView.colorizer.button*/currentBackgroundBrightness. Every
  existing property keeps its name and type.

## Qt5 fidelity verdicts (f0ad7b23 read directly)

- applyTheme (Qt5 ~60-128) and scheme (~159-191) bodies are
  byte-identical to HEAD's modulo the port's theme->plasmaTheme
  rename and added null fallbacks. The Qt5 body is the spec; no fork
  reinterpretation involved.
- The touching-window branch condition keeps JS precedence VERBATIM:
  `existsWindowTouching || existsWindowTouchingEdge && !(...active...)
  && activeWindowSchemeExists` - && binds tighter, so
  existsWindowTouching ALONE selects the current-screen
  active-window scheme. The comment above it ("when they are not
  active the active window scheme is used") suggests the intended
  grouping differs, but Qt5 shipped this exact precedence and the
  tree has never been the defect: preserved bit-for-bit, pinned by a
  named test case documenting the precedence, flagged here for a
  future deliberate decision.
- The Smart+Busy branch's `themeContrastedBackground` variable is
  dead in Qt5 and the port (computed, never read); not carried into
  the core (noted in the header comment).
- Deliberate divergences (each commented at the site):
  - the wrapper's enum->object mapping is total: a chosen source
    whose object is null yields schemeColors=null -> "kdeglobals" in
    the shell, where the Qt5 QML would have thrown a TypeError
    reading .schemeFile of null (latent, practically unreachable -
    layout schemes fall back to the kdeglobals SchemeColors object).
  - the edit-mode branch's console.log breadcrumbs move to qDebug in
    the wrapper (same three messages, same trigger).

## Eliminated invalid states (step-2.5 point b; for the ledger tick)

- "scheme file from nobody": SchemeFile is an enum class with an
  explicit KdeGlobalsFallback alternative - the "" / undefined string
  a missing themeExtended used to threaten is unrepresentable; the
  kdeglobals fallback is a named state, not a stringly default.
- "applyTheme aliasing guessed at": whether the chosen scheme IS the
  plasma default is a pure function of (source, isLightTheme) with
  the provider algebra proven from theme.cpp - the QML's repeated
  `=== plasmaTheme` object comparisons cannot drift apart anymore
  (single authority, one truth table).
- "out-of-range settings enums": the wrapper refuses unknown
  themeColors/windowColors ints loudly (qCritical) and falls back to
  plasma-default rendering; the core takes the real Q_ENUM types, so
  an invalid setting cannot silently walk the tree.
- "half-provided window tracker": the currentScreen facts
  (existsWindowTouching etc.) are only read behind
  windowsTrackerEnabled in the core, mirroring the QML guard; the
  shell binds them guarded, so no null-deref path exists.
